# 场景 3：出错 Debug（定位与修复）

## 输入模板（必填）

- 现象或报错（不会贴日志就描述现象）
- 大概怎么触发（不会复现可写“偶发/不稳定”）
- 你希望它怎么表现（不知道可写“和之前一样”）
- 其他已知线索（可空）

> 说明：用户不知道技术细节是正常情况。缺失信息由 AI 在排查过程中补齐。

## 执行流程（固定 6 步）

1. **先读 `goal.md` 和 `structure.md`，结合用户描述锁定 bug 大致位置**

- 明确业务目标、关键模块与可能受影响区域。
- 对缺失信息先做最小假设并标注“待验证”。

1. **结合具体代码，搜集证据并划定准确范围**

- 阅读相关文件与调用链，收集日志、报错堆栈、输入样例。
- 输出明确影响边界（模块/函数/数据路径）。
- 如用户无法提供日志/环境，AI 先构造最小复现并补齐关键上下文。

1. **给出假设原因并进行验证**

- 列出候选根因并逐项验证。
- 保留有证据支撑的根因结论。

1. **给出修复方案并列出修复计划写进 `plan.md`**

- 写清改动点、风险、验证方式与回滚思路。
- 必要时先与用户确认方案。

1. **实施修复并验证修复结果**

- 按 `rules.md` 编码修复。
- 验证问题用例与关键回归用例。

1. **根据修复经验更新 `rules.md`**

- 把本次修复沉淀为规范（禁用写法、推荐写法、检查项）。

## 输出物清单

- bug 定位记录（大致位置 + 准确范围）
- 根因验证记录（假设与证据）
- 修复方案与计划（写入 `plan.md`）
- 修复后验证结果（问题用例 + 回归用例）
- 更新后的 `rules.md`

## 完成判定

- [ ] 已基于 `goal.md` 与 `structure.md` 锁定大致位置
- [ ] 已形成证据链并划定准确范围
- [ ] 根因结论有证据支撑
- [ ] 修复方案与计划已写入 `plan.md`
- [ ] 问题修复并通过验证
- [ ] 已根据经验更新 `rules.md`
