# 场景 1：从零开始写代码

## 输入模板（必填）

- 要做什么（1 句话）：
- 最终要交付什么（程序/脚本/页面/接口）：
- 明确不做什么（可选，避免范围膨胀）：
- 怎么算完成（可验证，至少 1 条）：

## 执行流程（固定 5 步）

1. **需求澄清与边界确认**

- 对模糊需求提出澄清，形成明确问题清单。
- 判断属于 L1 / L2 / L3；默认按 L2。

1. **生成详细 `goal.md`**

- 写清目标、范围、成功标准、不做项。
- 若为 L1，小任务允许使用简版 `goal.md`，但仍需明确交付物与完成条件。

1. **生成详细 `plan.md`**

- 拆解里程碑、任务顺序、验收节点与风险。
- L1 至少写清 3 件事：要改什么、怎么验证、失败怎么办。

1. **用户确认门禁**

- L2/L3：与用户沟通并修改，直到用户同意 `goal.md` + `plan.md`。
- L1：若用户已明确要求直接交付，可在告知计划后直接实施，但不得跳过验证与结果说明。

1. **按计划实施编码 + 验证交付与文档回填**

- 严格依据 `plan.md` 执行，偏离需先更新计划。
- 验证运行结果与验收点。
- 仅在结构或可复用规则发生变化时更新 `structure.md` 与 `rules.md`。

## 输出物清单

- 已确认的 `goal.md`
- 已确认的 `plan.md`
- 代码实现与运行说明
- 验证记录（功能、边界、异常）
- 更新后的文档（如 `structure.md` / `rules.md`）

## 完成判定

- [ ] `goal.md` 已详细生成并经用户确认
- [ ] `plan.md` 已生成；若为 L2/L3，已获用户确认
- [ ] 能运行（主流程成功）或已提供不可运行原因与替代验证
- [ ] 输出符合“最终要交付什么”
- [ ] “怎么算完成”的条件全部满足
- [ ] 至少提供 1 条验证证据（命令、截图、日志、测试结果、样例输出其一）
